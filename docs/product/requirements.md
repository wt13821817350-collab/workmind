# WorkMind 产品需求文档

> 版本：v0.1 | 状态：草稿 | 日期：2026-07-16

## 第一章：项目背景（Background）

### 1.1 为什么要开发 WorkMind？

过去两年，AI 正在快速进入普通人的生活。ChatGPT、Claude、文心一言……越来越多的人开始尝试用 AI 工作。

但企业真正落地 AI，依然困难。

原因有四个：

**1. AI 工具太多，但没有一个能真正"干活"**

大多数 AI 工具只是"聊天框"。员工问一个问题，得到一段文字，然后自己再去执行。AI 扮演的是"建议者"，不是"执行者"。

**2. 企业的工作是流程，不是对话**

一个客服每天要处理几十条工单。一个运营每天要写推文、更新数据、回复用户。这些工作是有顺序的、有规则的、可以被系统化的。但现有的 AI 工具没有针对这种场景设计。

**3. 员工重复劳动严重，AI 却没有记忆**

企业知识分散在文件夹、群聊、邮件里。每一个新员工、每一个新项目，都要重新搜集信息。AI 本应成为企业的知识库，但缺少统一的接入方式。

**4. 中小企业买不起、用不起大企业的 AI 方案**

大企业有钱做定制系统。中小企业只能凑合用通用工具。它们需要一个足够专业，又足够易用的 AI 工作平台。

### 1.2 WorkMind 的出发点

WorkMind 不想做另一个 AI 聊天工具。

WorkMind 的目标是：**让 AI 成为企业里真正能干活的员工。**

## 第二章：市场分析（Market Analysis）

### 2.1 现有竞品

目前市场上有几类产品在做企业 AI 落地：

| 产品 | 类型 | 核心能力 |
|------|------|----------|
| Dify | AI 应用开发平台 | Workflow、RAG、API 接入 |
| Coze | AI Bot 平台（字节） | 插件系统、多 Bot 协作 |
| FastGPT | 知识库问答 | RAG、对话管理 |
| AnythingLLM | 本地知识库 | 私有化部署、文件问答 |
| MaxKB | 开源知识库 | 知识管理、API 调用 |

### 2.2 现有产品的问题

这些产品都有价值，但它们都有同一个问题：**更像 AI 工具，不像 AI 员工。**

具体来说：

- **偏聊天，不偏执行**：核心交互是对话框，用户问 → AI 答，缺少主动执行任务的能力
- **企业流程能力不足**：Workflow 配置复杂，非技术人员无法自助使用
- **Agent 能力有限**：更多是"调用工具"，缺少对业务语境的理解，难以胜任完整的工作任务
- **权限体系弱**：缺少角色权限、数据隔离、操作审计，不适合企业内部使用
- **知识与流程割裂**：知识库和 Workflow 是两个独立模块，没有打通

### 2.3 WorkMind 的机会

WorkMind 的机会，不在于"做一个更好的聊天 AI"，而在于：

**占据"企业 AI 员工"这个至今没有产品真正占领的位置。**

具体差异化方向：

1. **以"员工"为产品隐喻**：可以被分配任务、完成工作的数字员工，不是机器人，不是助手
2. **知识 + 流程 + 执行 三合一**：三者在同一平台里无缝配合
3. **面向非技术人员设计**：HR、运营、客服不需要懂 AI，就能搭建属于自己部门的 AI 员工
4. **适合中小企业**：支持 Docker 私有化部署，兼容任意 OpenAI 兼容模型

## 第三章：用户画像（User Persona）

WorkMind 的用户，不是"企业"。是企业里一个个具体的人。

---

### 用户 A：运营小李

**职位**：内容运营，互联网公司，2 年经验

**每天在做什么：**
- 整理昨天的数据报表，粘贴到群里
- 写推文或短视频文案，改了 3 遍还不满意
- 处理用户反馈，复制粘贴回复模板
- 统计活动效果，手动汇总 Excel

**痛点：**
- 写文案要翻历史文件夹找参考，找不到就重写
- 每次活动汇报都要从头整理，耗时 2 小时
- 大量工作是重复的，没有工具帮他自动化

**对 WorkMind 的期待：**
- 一个知道公司风格和历史内容的 AI，帮他起草文案
- 数据汇总自动跑，不用每次手动操作

---

### 用户 B：客服小王

**职位**：客服专员，电商公司

**每天在做什么：**
- 处理 100+ 条用户消息，大多数是重复问题
- 翻找产品手册回答用户问题，每次都要重新搜
- 按规则判断售后问题是退款还是换货
- 把当天工单整理成日报发给主管

**痛点：**
- 重复问题占 80%，没有智能工具自动回复
- 产品手册版本乱，找不到最新答案
- 日报手动写，下班时间全浪费在这里

**对 WorkMind 的期待：**
- 一个了解产品知识的 AI 帮他起草回复，自己审核发送
- 日报自动生成，看一眼确认就行

---

### 用户 C：HR 陈姐

**职位**：HR 专员，制造业，管理 200 人

**每天在做什么：**
- 整理招聘简历，手动筛选
- 回复员工问题：假期怎么算、社保怎么交
- 准备入职材料，每次都要找模板
- 组织培训，写通知、记录、总结

**痛点：**
- 常见问题每天都有人问，她每天都要重复回答
- 模板文件分散，每次找都要翻好久
- 招聘季工作量翻倍，根本忙不过来

**对 WorkMind 的期待：**
- AI 能基于公司制度文件回答员工常见问题
- 简历初筛自动跑一遍，她只看推荐结果

---

### 用户 D：销售老张

**职位**：大客户销售，传统制造业

**每天在做什么：**
- 整理客户拜访记录
- 写跟进邮件和方案
- 查库存、算折扣、准备报价单
- 月底整理销售数据，写月报

**痛点：**
- 跟进记录分散在备忘录、微信、邮件里，没有统一管理
- 每次写方案都要从零开始，参考上次的还要找半天
- 月报耗时，但领导不一定看

**对 WorkMind 的期待：**
- 客户跟进记录统一管理，下次拜访前自动提醒
- 邮件和方案草稿能基于客户历史自动生成

---

### 用户 E：行政小美

**职位**：前台兼行政，50 人小公司

**每天在做什么：**
- 接听电话，转接或登记
- 处理快递、采购、会议室预约
- 写各种通知和公告
- 帮老板整理文件、做表格

**痛点：**
- 杂事多，没有优先级，经常被各种小事打断
- 写通知格式不标准，老板总是改
- 文件管理混乱，找东西靠记忆

**对 WorkMind 的期待：**
- 通知和公告能按模板自动生成
- 公司文件有统一的地方管理，能搜索

---

### 共同规律

| 特征 | 描述 |
|------|------|
| 技术能力 | 低，不会写代码，不了解 AI 原理 |
| 工作类型 | 以重复性任务为主，有规律可循 |
| 最大痛点 | 找信息慢、写东西慢、重复劳动多 |
| 对 AI 的态度 | 愿意尝试，但怕复杂，怕出错 |
| 核心诉求 | 帮我干活，不要让我学新技能 |

## 第四章：业务场景（Business Scenarios）

这一章回答一个问题：**WorkMind 真正卖什么。**

不是功能列表，是真实的企业场景。每一个场景，都是一个真实的痛点，也是一个真实的销售理由。

---

### 场景一：AI 审批助手

**角色**：行政、HR

**现在怎么做：**
员工提交审批申请 → 行政手动查看内容 → 判断走哪个流程 → 通知相关负责人 → 等待签字 → 归档

**用 WorkMind 之后：**

```
员工提交审批
    ↓
AI 自动识别审批类型（报销 / 请假 / 采购 / 用印）
    ↓
自动填写标准字段，匹配审批模板
    ↓
通知对应负责人，附带原始材料
    ↓
审批完成，自动归档，通知申请人
```

**价值**：行政从"传达员"变成"监督者"，重复操作全部消失。

---

### 场景二：AI 知识助手

**角色**：HR、客服、运营

**现在怎么做：**
员工问问题 → HR 找制度文件 → 翻页查找 → 复制粘贴回答 → 员工再追问

**用 WorkMind 之后：**

```
管理员上传公司制度、产品手册、FAQ 文档
    ↓
AI 自动切片，建立企业知识库
    ↓
员工直接用自然语言提问
    ↓
AI 引用原文段落回答，标注出处
    ↓
知识更新时，重新上传即可同步
```

**价值**：企业知识不再锁在少数人脑子里，新员工第一天就能自助找到答案。

---

### 场景三：AI 工作流

**角色**：运营、数据分析

**现在怎么做：**
每天早上手动下载 Excel → 清洗数据 → 写日报 → 复制到微信群 → 重复 365 天

**用 WorkMind 之后：**

```
定时触发（例如每天 9:00）
    ↓
自动拉取数据源（Excel / API / 数据库）
    ↓
AI 分析数据，生成日报文本
    ↓
按模板格式化输出
    ↓
自动发送到指定群或邮件
```

**价值**：重复性数据工作彻底自动化，运营从"搬运工"变成"决策者"。

---

### 场景四：AI 数字员工

**角色**：行政

**现在怎么做：**
每天收邮件 → 判断类型 → 下载附件 → 手动录入系统 → 通知相关领导

**用 WorkMind 之后：**

```
邮件到达，AI 自动读取
    ↓
识别邮件类型（合同 / 发票 / 询价 / 通知）
    ↓
下载附件，OCR 提取关键字段
    ↓
按规则录入对应系统
    ↓
通知负责领导，附带摘要
```

**价值**：这不是"用 AI 辅助行政"，这是"AI 代替行政完成日常事务"。这是 WorkMind 最终要卖的东西。

## 第五章：产品定位（Positioning）

### 一句话定位

WorkMind 是一个面向企业办公场景的 AI 工作平台，通过知识管理、AI 工作流和数字员工，帮助企业自动完成重复性工作。

### WorkMind 不是什么

定位清晰，同样重要：

| 不是 | 原因 |
|------|------|
| 聊天机器人 | 不只是问答，要能执行任务 |
| AI 搜索引擎 | 不只是找信息，要能用信息干活 |
| 普通知识库 | 知识库是基础，不是全部 |
| 低代码平台 | 目标用户不是开发者，是业务人员 |
| RPA 工具 | 不是鼠标模拟，是 AI 理解 + 执行 |

## 第六章：产品目标（Goals）

### 用户目标

- 减少重复劳动，每人每天节省 1-2 小时
- 提高工作效率，减少人工出错
- 降低培训成本，新员工快速上手
- 统一企业知识，不再依赖个人记忆

### 企业目标

- 降低人工成本，减少因重复工作产生的岗位需求
- 提高办公效率，缩短业务流程处理周期
- 统一流程标准，减少因人员变动导致的流程断裂
- 降低出错率，规则执行由 AI 保障一致性

### 技术目标

- 模块化架构，每个功能独立可替换
- 插件化扩展，支持第三方工具接入
- 兼容 OpenAI 协议，不绑定单一模型厂商
- 支持 MCP 协议，面向未来 Agent 生态
- 支持私有化部署，保障企业数据安全

## 第七章：MVP 范围（MVP Scope）

MVP 的原则：**不求全，求能用。**

第一版只做最核心的闭环：用户能登录、能上传知识、能用 AI 对话、能配置简单工作流。

### 开发阶段规划

**第一阶段：用户基础**
- 用户注册 / 登录
- JWT 鉴权
- 角色权限（管理员 / 普通用户）

**第二阶段：知识库**
- 文档上传（PDF、Word、TXT）
- OCR 识别
- 文档切片 + 向量化
- 知识库管理

**第三阶段：AI 对话**
- 基于知识库的 RAG 问答
- Prompt 配置
- 对话历史记录
- 多模型切换

**第四阶段：工作流**
- 可视化 Workflow 编排
- 工具调用（HTTP / 数据库 / 文件）
- 定时任务触发
- 任务执行日志

**第五阶段：Agent**
- Agent 配置与管理
- MCP 协议接入
- 插件系统
- 多 Agent 协作（后续版本）

> 本文档第七章是开发优先级的依据，后续所有功能开发按此阶段推进。

## 第十一章：产品原则（Product Principles）

这一章不是需求，是约束。当团队在做取舍、在争论要不要做某个功能时，回到这里判断。

### 功能原则

所有功能，**必须**：
- 解决企业真实存在的问题

所有功能，**不能**：
- 为了 AI 而 AI —— 不因为"现在流行"就加上 AI，功能要先证明能解决问题
- 为了 Agent 而 Agent —— Agent 是手段，不是目的，简单的规则能做到的事不用 Agent
- 为了炫技 —— 复杂的技术方案如果不能让用户少做一步操作，就不该上线

### 模块原则

所有模块，**必须**：
- 可以复用 —— 一个模块要能被多个场景调用，不为单一场景写死逻辑

### 流程原则

所有流程，**必须**：
- 可执行 —— Workflow 设计出来不是用来看的，必须能跑通、能落地到真实业务

### AI 回答原则

所有 AI 回答，**必须**：
- 可追溯 —— 用户能看到 AI 为什么这么回答，答案从哪里来，而不是一个黑盒结论

### 知识原则

所有知识，**必须**：
- 有来源 —— 知识库里的每一条信息都能定位到原始文档，不能是模型自己"编"出来的
